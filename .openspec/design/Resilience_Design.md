# 畅捷通 Stream Gateway 可靠性与容错设计 v0.1.0

## 1. 极速自愈 (Fast Self-Healing)

网关系统的核心可靠性建立在"快速失败、快速剔除"的基础上。

### 1.1 路由主动剔除 (Active Eviction)
- **场景**: Node A 向 Node B 转发 Webhook，但 Node B 返回 `ECONNREFUSED`（节点进程崩溃）。
- **策略**: Node A 立刻从 Redis `route:{AppKey}` 中删除 Node B 对应的记录，无需等待 60s 的 TTL 过期。
- **目的**: 确保后续 Webhook 请求不再路由到已宕机的节点。

### 1.2 客户端高频心跳 (Heartbeat Challenge)
- **频率**: 10s Ping / 5s Pong。
- **逻辑**: 若 20s 未收到交互，网关主动 Close WS 连接并清理 Redis 路由。
- **重连**: ISV 客户端执行指数退避重连，并携带 Jitter 打散重连请求，防止"惊群效应"。

## 2. 背压与流量控制 (Backpressure)

网关作为一个挂起请求的代理，内存消耗取决于当前活跃的 Webhook 数量。

### 2.1 内存级并发保护 (Node-Level)
- **并发上限**: 单机限制 5,000 个挂起请求。
- **行为**: 达到上限后，新的 Webhook 直接返回 HTTP 503。Core 会在随后衰减重试。

### 2.2 租户级并发限流 (Tenant-Level)
- **并发上限**: 单 AppKey 限制 100 个挂起请求。
- **目的**: 防止某个故障的 ISV 客户端（如处理极其缓慢）占满网关所有连接池。
- **行为**: 达到上限后返回 HTTP 429。

## 3. 故障切换场景 (Failover Matrix)

| 故障场景 | 现象 | 自动恢复逻辑 |
| --- | --- | --- |
| **网关节点 A 宕机** | 与 A 连接的客户端全断开 | 客户端通过 LB 漂移到节点 B。A 留下的 Redis 路由随 TTL 到期或被其他节点主动剔除。 |
| **Redis 集群宕机** | 无法读写路由表 | 网关降级使用节点内存中的"最近路由缓存"。无法处理新 AppKey，但能维持现有连接的转发。 |
| **ISV 客户端假死** | 不响应 WS 消息 | 网关 3s 后触发 ACK 超时，返回 Core 504。随后 20s 心跳超时断开连接。 |
| **P2P 转发超时** | 节点间通信异常 | 转发节点返回 503，Core 自动重试，下次请求可能打到不同节点。 |

## 4. 熔断机制 (Circuit Breaker)

当某个 AppKey 的 Webhook 转发失败率（或超时率）在 1 分钟内达到阈值（如 50%）：
- **状态**: 进入 Open 状态。
- **行为**: 网关在入口处直接拦截该 AppKey 的 Webhook，返回 503。
- **持续时间**: 60s 后尝试 Half-Open，放行少量请求观察。
