# 提案：SDK 智能重连策略与“就绪待命”模式优化 (v1.0)

**提案编号**: `SYKFPT-1061-Patch-SDK-Resilience`
**状态**: 已实施 (Archived)
**目标模块**: `connector-sdk-java`

---

## 1. 问题背景 (Problem Statement)

目前网关对单租户（AppKey）设置了最大并发连接数限制 ($N$)。当 ISV 部署了超过 $N$ 个实例（即 $N+M$ 个）时，系统面临以下风险：
- **补位真空 (Slot Vacancy)**：当处于在线状态的 $N$ 个节点中某一个发生崩溃（Crash）时，剩余处于待命状态的 $M$ 个节点由于可能处于**长周期的指数退避 (Exponential Backoff)** 阶段（最高可达 60s），无法在秒级感知并抢占空出的槽位。
- **重试步调一致 (Step Synchronization)**：$M$ 个待命节点若采用完全一致的退避算法，会产生周期性的“惊群效应”，瞬间冲击网关。

## 2. 核心解决方案：就绪待命模式 (Standby Mode)

SDK 的重连策略具备**错误语义感知**能力，将“因故障导致的退避”与“因排队导致的待命”逻辑解耦。

### 2.1 状态机演进 (State Machine)

| 状态 | 描述 | 下一触发动作 |
| :--- | :--- | :--- |
| **INITIAL** | 初始启动 | 立即执行握手 |
| **CONNECTED** | 握手成功 | 维持心跳与业务处理，稳定 60s 后重置计数 |
| **BACKOFF** | 网络/环境故障 (500/Timeout) | **指数退避**：$wait = min(60s, base 	imes 2^n)$ |
| **STANDBY** | **连接数已满 (503/429)** | **就绪抢占**：$wait = Base(5s) + Random(0, 10s)$ |
| **STOP** | 鉴权失败 (401/403) | **停止重连**，避免死循环 |

### 2.2 关键逻辑设计

#### A. 竞争性随机抖动 (Competitive Jitter)
在 **STANDBY** 状态下，SDK 放弃“翻倍等待”逻辑，改为执行一个**短周期的随机波动圆环**。
- **退避上限锁定**：重试间隔锁定在 **5~15秒** 之间。
- **目的**：确保一旦 $N$ 出现空缺，待命节点能在极短时间内发起握手，且 $M$ 个节点间的请求压力被随机均匀打散。

#### B. 提升可测试性 (Testability)
为了在 JDK 21 环境下顺利进行 Mock 测试（规避 Byte Buddy 对 JDK 21 内部类增强的限制），引入了 `IHttpProvider` 接口来封装原生的 `HttpClient` 调用。

## 3. 实施成果 (Implementation Results)

1.  **高可用自动补位**：待命节点能以平均 7.5s 的速度填补失效槽位，保障 ISV 的业务连续性。
2.  **集群流量削峰**：随机抖动机制（Jitter）防止了大规模失效后的并发冲击。
3.  **单元测试覆盖**：重构了 `GatewayClient`，完善了相关逻辑的单元测试并 100% 通过。
