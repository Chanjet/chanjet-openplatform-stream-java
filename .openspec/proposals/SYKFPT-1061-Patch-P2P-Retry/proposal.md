# OpenSpec 补丁提案：P2P 转发重试机制 (SYKFPT-1061-Patch-P2P-Retry)

| 属性 | 内容 |
| --- | --- |
| **提案 ID** | SYKFPT-1061-Patch-P2P-Retry |
| **状态** | 草案 (Draft) |
| **负责人** | Gemini CLI |
| **优先级** | 高 (High) |

---

## 1. 问题描述 (Problem)
在分布式集群环境下，当客户端发生连接漂移（从节点 A 切换到节点 B）时，Redis 中的路由信息可能存在短时延迟或“鬼影路由”。如果 Webhook 请求到达节点 A，且节点 A 根据过期路由尝试向已失效的节点发起单次 P2P 转发，一旦该转发失败（连接超时、404 或目标节点宕机），消息将直接丢失。

## 2. 解决方案 (Solution)
- **迭代尝试策略**: 修改 `MessageDispatcher`，使其在 P2P 转发失败时，不立即抛出异常，而是从该 AppKey 的所有可用路由中排除掉刚刚失败的节点，并尝试转发给下一个候选节点。
- **最大重试限制**: 为了防止网络风暴和长链等待，设置最大重试次数（建议为 2 次）。
- **环路检测**: 确保转发请求不会在节点间形成无限循环（例如 A -> B -> A）。

## 3. 实施计划 (Implementation Plan)
1.  **重构分发逻辑**: 更新 `MessageDispatcher.doDispatch` 方法，引入循环尝试逻辑。
2.  **增强 P2P 客户端**: 使 `IP2PClient` 能够返回明确的发送状态（成功/失败）。
3.  **Red (红)**: 编写模拟测试，模拟一个 AppKey 对应两个 Node，其中一个 Node 无法连接，验证系统能否最终通过另一个 Node 成功下发。

## 4. 验证策略 (Verification Strategy)
- 启动三个实例，手动在 Redis 中插入一个不存在的 Node ID 路由，验证 Webhook 请求能否在跳过错误节点后成功通过真实节点下发。
