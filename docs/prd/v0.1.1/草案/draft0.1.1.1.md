# Openplatform CLI tool (Stream Connector) 需求文档草案

## 1. 产品定位与核心功能
**定位**：一个支持后台常驻运行的命令行工具（CLI），兼具**畅捷通开放平台 API 调试/调用**、**流式事件监听及 Webhook 转发**、以及**本地透明 HTTP 代理**的三重能力。
**生态外延**：作为数字员工生态的一环，本工具的副产物是一份标准化的 **AI Agent Skill（大模型技能库）**，使大语言模型能够原生理解、安装并代理执行本 CLI 的交互，实现“人机共驾”。

### 1.1 核心功能列表
1. **API 便捷调用**：通过 CLI 命令获取并调用畅捷通开放平台的 API。
2. **凭据与会话管理**：自动维护 `appKey`、`appSecret`、`encryptCode`、自建应用证书、`openToken`、`appTicket` 等认证信息，并支持 `openToken` 的自动轮转刷新。
3. **流式事件监听与转发**：由底层的 **Connector Server** 统一提供 Stream 模式的消息对接方式，CLI 借助此通道实时监听开放平台推送的事件，并支持安全转发至客户指定的 Webhook 地址。
4. **本地透明 HTTP 代理**：开启一个仅局域网内（或仅本进程间）可访问的 HTTP 端口代理，让其他难以接入签名的老旧脚本/微服务以“零鉴权”形式安全地呼叫开放平台。
5. **多实例支持**：允许在同一台机器上同时运行多个 CLI 实例，以独立管理和监听不同的 `appKey` 租户。
6. **开机自启**：支持在宿主机启动时自动唤起 CLI 后台应用，保障事件监听及代理服务的高可用。
7. **审计与合规**：支持在本地保留操作审计记录、API 调用流水以及 Webhook 转发失败时的异常记录。

## 2. 详细命令与交互全景

> **全局多实例路由与别名参数 (`--profile`)**：
> 鉴于 CLI 架构原生支持多实例并发运行，所有的命令行操作（无论 `api` 调用、`status` 查看还是 `config` 修改）均需提供一个统一的归属锚点。
> - 命令默认作用于初次初始化的主体实例（系统默认别名为 `default`）。
> - **个性化别名映射**：为了消除记忆长串 `appKey` 的负担，CLI 允许在 `init` 或配置阶段为每一个应用实例设定一个易读的**环境别名 (Alias)**（例如 `shop-A`、`uat-env` 等）。
> - **跨实例精准路由**：当需要操纵其他租户环境时，只需附加对应的别名参数（如 `cli api get /v1/orders --profile shop-A`），即可使后续命令无缝切入对应的独立实例上下文中执行。

> **全局命令行向导 (`--help` / `help`)**：
> 提供 CLI 根级别的全局帮助视图。当用户在最顶层输入 `cli --help` 时，终端将以结构化的方式输出工具支持的所有一级子命令清单（如 `init`, `api`, `proxy`, `webhook`, `status` 等）及其一句话指引，同时列出公共的全局环境参数（如 `--profile`），充当开发者总览全局的“操作大纲”。

### 2.1 初始化与配置管理 (`init` & `reset`)
- **首次初始化流程 (`init`)**：
  - **前置应用创建引导**：在要求用户输入凭证前，通过交互式 Prompt 优先询问用户是否已经在开放平台后台“创建了自建应用”。若用户表示“未创建”或不知道如何获取凭据，CLI 则必须打印极其清晰的“自建应用创建与取参指南”（包含控制台网址链接、核心步骤流）。必须强制要求用户确认“已创建”并且拿到了关键参数，方可进入真实的配置初始化。
  - **凭据拉取录入**：当确认已建好应用后，依次引导录入 `appKey`、`appSecret`、`encryptCode`、自建应用证书等。
  - **灵活交互闭环**：除了向导式交互外，也支持纯命令行参数注入赋值（适合脚本自动化，默认跳过前置创建确认）。
  - > [!CAUTION]
    > **安全约束**：敏感凭据必须在本地进行安全加密存储，防范跨权限非法读取。由于安全考量，**不提供任何形式的凭据“一键导出/便捷迁移”功能**。

- **重置状态**：清除当前实例的所有凭据和运行时状态，恢复为空白安装状态以供重新初始化。

### 2.2 API 调用体验 (`api`)
针对 OpenAPI 规范中 `tags` 和 `operationId` 非必填的现状，为保证所有接口均能无缝透出，CLI 采用最直观的 **`Method + Path` 范式**设计调用命令：
**命令结构**：`{cliName} api <METHOD> <PATH> [flags]`

- **全景接口透视 (`list`)**：
  - 提供专属的 `list` 子命令用于查询当前可用的 API 列表。**此可用 API 文档及列表全景树将直接通过畅捷通开放平台的通道获取**，获取这些数据的关键凭证参数为用户的 `appKey`、`appSecret` 以及自建应用证书。
  - **渐进式明细**：默认执行 `cli api list` 时，终端仅简洁清爽地按行打印**命令调用串 (Method+Path) 及对应的一句话描述**；只有当用户显式增加详尽参数（如 `cli api list --detail`）时，CLI 才会将各个 API 的参数级详情（包含必填/非必填项及类型）平铺展开渲染。
  - **内嵌语义检索 (Semantic Search)**：区别于传统的死板正则/关键字过滤，`--search` 参数将被赋予更为智能的灵魂。计划**在 CLI 内部轻量级挂载 Embedding 算法及单机向量数据库**（如 SQLite-VSS / faiss）。当用户或 Agent 输入 `cli api list --search "我想新建一笔收款记录"` 时，系统能超越字面束缚，基于向量空间计算语义相似度，精准召回所需的最贴切的底层 API。
- **动态反向匹配（所见即所得）**：用户直接输入真实的请求路径（如 `cli api get /v1/orders/888`），底层 CLI 引擎会自动反向匹配 OpenAPI 中的模板路径 `/v1/orders/{orderId}`，提取对应 Path 参数（`orderId=888`）并完成请求组装。
- **单接口详尽手册 (`--help`)**：支持对单一具体的 API 节点靶向查询使用说明。当在具体路径后附加帮助参数（如 `cli api get /v1/orders --help`）时，CLI 需直接输出专属于该接口的**完整、参数级明细的手册**，清晰且详实地展示该接口所有支持的传入参数（Query/Header/Body 的字段名、必填项限制、数据规范及枚举字典等）。
- **参数平铺与强类型校验**：Query、Header 参数采用扁平的 `--flag` 注入，JSON Body 需灵活支持行内文本 (`--body '{"a":1}'`) 或挂载文件读取 (`--body @data.json`)。在发起网络调用前，按照 OpenAPI 定义的 Schema 强制在本地拦截参数错误。
- **无感认证执行**：用户调用 API 期间彻底屏蔽一切认证细节，自动注入签名与 Token。

### 2.3 本地透明 HTTP 代理 (`proxy`)
- **零代码门槛接入生态**：通过启动一个监听绑定在 `localhost / 127.0.0.1` 的代理端口（例如 `http://127.0.0.1:8080/v1/...`），使本机的其他脚本应用无需实现任何签名算法、也不必读取 `appKey/appSecret`。
- **背靠背转发**：应用将原生 API 调用向该本地端口抛出后，由 CLI 后台实例完整接管身份编排、报文加密签名、Token 挂载等逻辑。
- **代理安全防范**：
  - > [!CAUTION]
    > **越权隔离**：为了防止宿主机上发生恶意探测或外部网络扫描导致的越权 API 攻击，此代理服务**绝不应暴漏给公网（0.0.0.0）**，必须强行锁定监听在 `127.0.0.1` 环回安全接口，或加入自定义的轻量化本地令牌鉴权（Local Auth Headers）。

### 2.4 Webhook 与流式监听 (`webhook`)
- **配置与鉴权过滤**：配置目标 Webhook 地址后，后台引擎将监听到的事件（除 `appTicket` 等受限敏感消息外）透明转发。
- **告警与异常兜底**：
  - > [!WARNING]
    > **异常链路**：若 Webhook 接收端宕机、网络异常或返回报错，工具必须将失败事件**单独隔离并持久化记录**，同时应支持设定重试策略或离线告警通知机制。

### 2.5 日志治理与可观测性 (`log`)
为了满足排查调试、安全审计与事件兜底等不同诉求，CLI 后台应用必须将日志进行**清晰的分类管控**，具体输出类别如下：
1. **系统运行日志 (System Log)**：记录 CLI 常驻进程自身的生命周期（启停、崩溃重启）、`openToken` 续期结果、配置加载结果，以及后台本身的内部报错信息。
2. **调用流水/审计日志 (Access/Audit Log)**：记录用户通过 `api` 命令行以及 `proxy` 代理发起的所有向外 API 请求动作。包括请求时间、Method、URI、状态码以及脱敏后的耗时，用于事后审计“谁、在什么时间、请求了什么”。
3. **流式事件日志 (Event Log)**：当客户配置了“监听本地写入”而非 Webhook 转发时，开放平台推送过来的业务事件（Payload）将专门记录于此，作为本地存储的主凭据。
4. **死信与异常告警日志 (Dead-Letter Log)**：专门用于记录 Webhook 转发失败、或因为某些系统异常导致未能消费的事件数据，确保存档兜底以待后续手工重传或报警修复。

- **自定义挂载**：支持用户显式指定上述各分类日志的落盘绝对路径。若未配置，默认在 `~/.{cliName}/log/{类别}` 下自动生成独立通道日志。
- **灵活的滚动策略**：
  - > [!TIP]
    > **磁盘保护**：为了防止全量写入撑爆宿主机磁盘，上述四大类日志均需分别提供**独立的日志滚动截断配置项**（如支持全局/分类配置 `max-size=500MB` 或 `max-age=7d` 等滚动清理方案），确保运维安全。

### 2.6 集群状态与运维 (`status` & `config`)
- **运行状态检查**：展示 CLI 后台进程的存活状态、且**全面兼容多实例场景**下的各实例健康度查询。
- **配置清查编辑**：类似 `vim`/`nano` 模式阅览并编辑当前实例的配置，强制**脱敏**关键凭证（例如仅展示 `appSecret: a4b3********5c`）。
- **生命周期干预**：提供重启与关闭后台进程的命令（需设计 Graceful Shutdown 优雅停机保证）。
- **无缝升级探测**：
  - 提供 `check-update` 命令，通过服务端核验提示用户当前版本是否有新版本可供升级及升级指引。

## 3. 核心交付物二：AI Agent Skill (应用生态扩展)
除了供人类直接敲击使用的二进制文件（CLI）外，本产品的第二交付物是一套规范的**大模型工具描述集（Skill）**。

- **1. Skill 定位**：一份符合主流大模型规范（如 OpenAI JSON Schema / Markdown instructions）的文件集合，随 CLI 同步发布。
- **2. 赋能场景**：
  - **引导式零门槛安装**：用户遇到不懂安装编译的问题时，Agent 加载此 Skill 后可直接为其拼装运行 OS 相关的拉取安装命令。
  - **伴随式业务初始化**：依托上文 `2.1` 节的逻辑，由 AI 直接根据报错或缺失的凭证字段，解答“到底去哪找 appSecret”等帮助引导。
  - **全托管的自然语言调用**：将底层的 API Method+Path 暴露为自然语义节点。用户只需在对话框输入“帮我拉取昨天退款的订单”，Agent 即能基于内部的 Skill 知识，自动组织翻译为 `cli api get /v1/orders --status REFUND ...` 并在后台静默调度，最后只返回加工好的人类可读报表。

## 4. 面向 Agent 首选 (Agent-First) 的底层设计准则
如果 CLI 的一等公民是像 openClaw 这样的 Autonomous AI Agent，那么它就不仅是一个“人类工具”，而必须遵循以下 **Agent-Friendly** 工程准则，才能根治大模型调用工具时的“死锁阻塞、幻觉编造、解析错乱及上下文溢出”等痛点：

### 4.1 绝对的非交互防死锁机制 (Non-Interactive)
- **痛点分析**：Agent 在终端里遇到预期外的 `Press Enter to continue...`、`Vim 编辑器挂起` 或 `(Y/n)` 选择时，极其容易陷入无限阻塞的死锁（无法输入按键）。
- **底线设计**：全局所有命令、所有原本的交互式引导（涵盖 `init` 步骤和 `config` 修改），必须 100% 能够通过平铺式的命令行 `--flag` 参数、环境变量（ENV）或者标准输入流覆盖填入；遇到不可逆操作亦可追加强制确认参数（如 `--yes`）直接放行。

### 4.2 机器可读的硬核结构化输出 (Machine-Readable Output)
- **痛点分析**：传统的 CLI 习惯用颜色（ANSI Escape Codes）、ASCII 进度条或对齐表格等元素取悦人类。这对大模型的上下文提取而言就是一堆杂牌乱码，导致极高解析成本和错乱。
- **底线设计**：全局提供一个强级别的 `--format json` 输出选项。一旦挂载该参数，无论是 `list` 查列表、`status` 看状态信息，还是一条极其复杂的 Error 堆栈报错信息，CLI 必须全部回吐出拥有严谨单一层级 Schema 的纯 JSON 字符串。这能让 Agent 精确获取任何自己所需的 Key 字段并向后流转。

### 4.3 “试运行”防幻觉与自解释错误 (Dry-Run & Structured Feedback)
- **痛点分析**：遇到陌生的 API，大模型往往喜欢发挥“幻觉”，凭空捏造虚无缥缈的 `--query` 或者假想出 API Path 给发到线上接口去。
- **底线设计**：
  - **纯本地校验：** 提供 `--dry-run` 选项，让 Agent 提前向底层路由和 Schema Checker 发起注入；系统只去比对当前的入参结构对不对，并不真实地发出 HTTP 网络请求。
  - **带建议的报错：** 若命中参数错误（如缺失或类型不对），抛弃只说 `[Error] Parameter Mismatch` 的懒惰方式。报错 JSON 中必须携带具体的纠错建议和修正结构（犹如对 Agent 说出“你的入参有误，因为缺失了订单 ID 字段，请补齐后重试”）。由于 LLM 极强的问题修复自省能力，收到详实的 Error Report，它马上就能利用 `Self-Correction` 进行下一次成功的调用纠偏。

### 4.4 短上下文检索与强制截断保障 (Context Window Protection)
- **痛点分析**：很多工具把所有的命令选项或开放平台的 1000 个 API 接口，在一个 `cli api list` 里无保留地“吐给”终端。过大的输出体瞬间就耗尽了主流 LLM 几万、十几万 Token 的 Context Window 导致幻觉雪崩。
- **底线设计**：
  - 对于长列表命令必须执行默认的**暴力长尾截断策略**（比如一次至多输出 20 个）。
  - 给 Agent 提供充足的精确检索选项作为搜索手段（附加 `--search` 参数），强迫 Agent 采用“查字典”的短、频、快分段提取策略。
  - **【高阶护城河】：基于 Embedding 的高容错召回**：Agent 的提问往往是“业务语义”而非“标准接口名”。当出现指代不一致如 (搜索 “注销” vs OpenAPI 文档写着 “销户”) 时，纯文本 Search 会导致 Agent 找不到接口而放弃工作。内嵌向量检索 (`Semantic Search`) 将使得 Agent 不必强行“猜”准确规范词，只要发送其自身的长句 Prompt，即可稳定地从本地向量空间得到最关联的 Top 3 APIs，以保障业务不中断。
