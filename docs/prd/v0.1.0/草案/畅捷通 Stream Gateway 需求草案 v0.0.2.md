# 畅捷通 Stream Gateway 需求草案 (v0.0.2)

这是根据我们多轮深入讨论整理出的**畅捷通 Stream Gateway 需求草案 v0.0.2**。

相比 v0.0.1，本版本重点强化了**多客户端负载均衡、基础设施解耦、极致自愈机制**以及**面向未来的微服务演进能力**。

---

## 1. 背景与产品定位

### 1.1 核心痛点

*   **网络限制**：ISV 接入 Webhook 必须具备公网 IP 和 SSL，内网/本地环境接入困难。
    
*   **安全风险**：开放公网端口存在被攻击风险。
    
*   **接入复杂**：Webhook 签名校验与 URL 自动验证对开发者门槛较高。
    

### 1.2 产品定位

作为畅捷通开放平台的官方基础设施组件，提供 **Webhook-to-WebSocket 透明同步桥接** 能力。实现“免公网 IP、免证书、高安全、低延迟”的事件订阅体验。

---

## 2. 核心架构设计 (Core Architecture)

### 2.1 同步阻塞桥接模式 (Synchronous Bridge)

*   **无状态代理**：网关不设消息队列，不持久化业务数据。
    
*   **状态回传**：网关挂起 HTTP 请求，等待 WebSocket 客户端的 ACK。
    
*   **压力回传**：若客户端离线或处理超时，网关向畅捷通 Core 返回 `503/504`，利用上游原生的 **Webhook 衰减重试机制** 实现消息积压与重补。
    

### 2.2 P2P 动态路由机制

*   **路由注册表 (Redis)**：使用 Redis 存储 `AppKey -> Node_IP_Set` 的映射（TTL 60s）。
    
*   **分布式路由**：
    
    1.  Webhook 随机打到任一 Node。
        
    2.  该 Node 查询 Redis 获知客户端连接所在的物理节点。
        
    3.  通过内部 P2P HTTP 转发报文至目标节点。
        
*   **分散接入 (Dispersed Access)**：建议 ISV 部署多个客户端实例，通过基础设施负载均衡分散连接到不同网关节点，消除单点故障。
    

---

## 3. 负载均衡与分发策略

### 3.1 职责划分

*   **基础设施层 (L4/L7 LB)**：负责客户端（Connector）到网关节点（Node）的物理连接分发，建议采用 **Least Connections** 策略。
    
*   **应用逻辑层 (Gateway Internal)**：负责消息到具体客户端实例的逻辑分发。
    

### 3.2 双层公平分发算法

1.  **跨节点分发 (Node-to-Node)**：当 Webhook 到达非目标节点时，网关从 Redis Set 中**随机/轮询**选择一个目标 Node 进行内部转发。
    
2.  **节点内分发 (Node-to-Client)**：当目标 Node 本地存在多个属于同一 AppKey 的连接时，采用 **Round-Robin (轮询)** 算法选择一个连接进行推送。
    

---

## 4. 可靠性与自愈机制 (Resilience)

### 4.1 极致自愈 (Self-Healing)

*   **见死即埋 (Active Eviction)**：内部转发若遇连接拒绝（ECONNREFUSED），转发节点**立刻删除** Redis 中的失效路由并返回 503，不等待 TTL 过期。
    
*   **心跳挑战 (Health Check Challenge)**：客户端每 10s 发起 Ping。若 20s 未收到网关 Pong，立刻断开并重新发起建连（覆盖旧路由）。
    

### 4.2 优雅降级与背压控制

*   **内存级并发保护**：
    
    *   **节点级**：限制单机总挂起请求数（如 5000），防止 OOM。
        
    *   **租户级**：限制单 AppKey 在单机并发数（如 100），防止“吵闹邻居”干扰。
        
*   **宏观熔断 (Redis)**：若某 AppKey 连续超时/失败达阈值，在 Redis 标记熔断（60s），最外层入口直接拦截，保护集群。
    

---

## 5. 模块解耦与演进设计

### 5.1 逻辑分离，物理合一

*   **代码模块化**：严格分离 `HttpReceiver` (HTTP接收)、`WsManager` (WS管理) 与 `Router` (路由路由) 逻辑。
    
*   **对称部署 (Phase 1)**：初期采用全功能节点模式，资源利用率最高。
    
*   **非对称部署 (Phase 2)**：未来流量剧增时，通过启动参数拆分为“无状态 HTTP 负载层”与“重状态 WS 连接层”，实现微服务化平滑演进。
    

---

## 6. 接口与规范 (Specifications)

### 6.1 安全原则

*   **不解密原则**：网关仅转发 `Headers` 和 `Raw Body` 字符串。
    
*   **建连鉴权**：`sign = HMAC_SHA256(AppKey + Timestamp, AppSecret)`。AppSecret 不在网关落地。
    

### 6.2 客户端 (ISV) 开发规范

*   **快速 ACK**：必须在 3 秒内完成“接收 -> 验签 -> 本地落盘 -> 发送 ACK”。严禁同步执行耗时业务。
    
*   **强幂等性**：必须基于畅捷通原始 `Event-ID` 或业务单据号实现业务幂等处理。
    
*   **指数退避重连**：断线重连需包含随机抖动（Jitter），防止惊群效应。
    

### 6.3 自动化校验

*   **URL 自动激活**：网关自动处理畅捷通 Core 的 `GET check_code` 验证请求，无需 ISV 手动编写验证逻辑。
    

---

## 7. 运维与可观测性

*   **全链路 Trace**：记录 `trace_id` 在网关内的流入、转发、推送、ACK 全生命周期日志，不记录业务明文。
    
*   **监控维度**：单 AppKey 并发数、平均处理耗时 (Latency)、重试成功率、节点活跃连接数。
    

---

**版本说明**：v0.0.2 版本确定了应用层路由与基础设施 LB 的边界，确立了内存+Redis 混合背压机制，并为未来的服务分离预留了平滑通道。

### 图二：双层负载均衡消息流 (Double-Layer Routing Logic)

**核心点**：展示消息如何通过“跨节点”和“节点内”两层筛选，最终到达具体客户端。

```text
[ Webhook 到达 Node A ]
          │
          ▼
[ 第一层：跨节点路由 (Cross-Node) ]
    查 Redis: route:App123 -> Set {Node_A_IP, Node_B_IP}
          │
          ├─ (A) 命中本地连接? ──▶ 继续下行
          │
          └─ (B) 命中远程节点? ──▶ 随机选 Node B ──▶ [ P2P 转发至 Node B ] ──▶ (Node B 重复下行逻辑)
          │
          ▼
[ 第二层：节点内分发 (Intra-Node) ]
    查本地内存: App123 -> List {WS_Conn_1, WS_Conn_2}
          │
          └─ 轮询算法 (Round-Robin) ──▶ 选中 WS_Conn_2
          │
          ▼
[ 阻塞下发与 ACK ]
    1. 推送报文给 WS_Conn_2
    2. 挂起 HTTP 请求，等待 ACK
    3. 收到 ACK ──▶ 释放 HTTP 200 返回给 Core

```

### 图三：自愈与背压流程 (Self-Healing & Backpressure)

**核心点**：展示“见死即埋”和“内存限流”的防御细节。

```text
[ Webhook 请求 ] ──▶ [ 流量入口 ]
                          │
      ┌───────────────────┴──────────────────┐
      ▼ (1. 宏观熔断检查)                     ▼ (2. 内存背压检查)
 [ Redis 熔断位? ] ──(Yes)──▶ [ 返回 503 ]  [ 内存并发 > 阈值? ] ──(Yes)──▶ [ 返回 429 ]
      │ (No)                                 │ (No)
      ▼                                      ▼
 [ 执行内部 P2P 转发 ]                  [ 执行 WS 推送 ]
      │                                      │
      ├─(A) 目标节点拒绝 (ECONNREFUSED)       └─(C) 客户端 ACK 超时 (4s)
      │     │                                      │
      │     ▼                                      ▼
      │  [ 立刻从 Redis 删除该 Node 路由 ]     [ 释放请求，返回 504 ]
      │  [ 返回 503，触发 Core 重试 ]          [ 触发 Core 重试 ]
      │
      └─(B) 转发成功，等待 ACK...

```

### 💡 图示要点总结 (Key Takeaways)

1.  **分散连接 (Dispersed Connections)**：
    
    *   通过基础设施 LB 将多个客户端（Client 1, 2）分散到不同网关节点（Node A, B）。
        
    *   **优点**：消除单点故障，Node A 挂了，LB 和 Redis 会自动把流量切到 Node B。
        
2.  **两级负载均衡 (Two-Tier LB)**：
    
    *   **一级**：网关节点间根据 Redis 记录随机选目标 Node（跨机转发）。
        
    *   **二级**：目标节点内根据本地内存列表轮询选 WS 连接（同机分发）。
        
3.  **同步阻塞 (Sync Blocking)**：
    
    *   整个链路是“牵一发而动全身”的阻塞流。只有客户端 ACK 了，最前面的 HTTP 请求才算完成。
        
    *   **优点**：利用畅捷通 Core 已有的重试机制，网关实现“完全无状态”。
        
4.  **架构演进 (Phased Evolution)**：
    
    *   目前的 `Node` 既跑 `HttpReceiver` 也跑 `WsManager`。
        
    *   **未来**：只需调整 LB 策略，即可将 Node 群拆分为纯 HTTP 接入层和纯 WS 连接层，代码无需重写。
        

这几张图涵盖了从**物理部署**到**逻辑算法**，再到**异常处理**的全貌，是 v0.0.2 需求草案的核心视觉支撑。
