## 1. 产品定位与支持场景

提供 **Webhook-to-WebSocket 透明同步桥接** 能力。

- **内网接入**: ISV 无需公网 IP 和 SSL 即可接收业务事件。
- **高可用投递**: 结合 Core 原生衰减重试与网关 P2P 转发重试机制。
- **高性能并发**: 基于 Java 21 虚拟线程，支撑海量 ISV 连接。

---

## 2. 核心业务流程 (同步阻塞模式)

1. **Webhook 接收**: Node A 收到 Core 发送的 Webhook (POST)。
2. **本地优先分发**: Node A 检查本地连接。若命中，直接通过 WS 推送。
3. **集群 P2P 转发**: 若本地无连接，查询 Redis 路由表，将请求转发至持有连接的 Node B。
4. **重试机制**: 若 Node B 转发失败，Node A 自动尝试集群内其他候选节点。
5. **ACK 回传**: ISV 客户端返回 ACK。网关释放挂起的 HTTP 请求，返回 200 OK 给 Core。

---

## 3. 握手与安全协议 (No-Secret)

### 3.1 Nonce 申请
- **接口**: `GET /v1/ws/challenge?app_key={AppKey}`
- **鉴权**: 必须携带 `X-CJT-PreAuth` (16位 HMAC 前缀)。
- **逻辑**: 网关代理 Core `verify-preauth` 接口验证身份。

### 3.2 WebSocket 建连
- **URL**: `/connect?app_key=...&nonce=...&sign=...&client_id=...`
- **逻辑**: 网关代理 Core `verify-sign` 接口验证完整签名。

---

## 4. 容忍期自愈逻辑 (State Machine)

1. **触发**: Webhook 到达且 AppKey 全量离线。
2. **503 阶段 (30min)**: 网关持续返回 503，Core 执行衰减重试。
3. **禁用推送**: 离线超过 30 分钟，网关调用 Core 接口挂起推送（消息入离线池）。
4. **恢复推送**: 任一客户端重连，立即通知 Core 恢复实时推送并补发。

---

## 5. 分布式边界约束

- **本地优先**: 优先利用本节点资源，最大化降低 P2P 网络开销。
- **环路保护**: P2P 跳数限制为 1，防止极端路由错误下的无限转发。
- **令牌更新**: 支持 `internal-tokens` 列表，保障集群滚动更新时的互信。
