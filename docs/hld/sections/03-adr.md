# 架构决策记录 (ADR)

## [ADR-01] 共享存储后端选型 (Shared Storage Backend)
- **背景**: Cowen 需要支持多种主流关系型数据库以适配不同 ISV 的基础设施。
- **选型对比**:
  - **方案 A (SQLx - 异步、原生 SQL)**:
    - 优点: 完全异步支持 (Tokio), 无需额外运行时, 编译时 SQL 校验, 适配多种 DB (MySQL/PG/SQLServer)。
    - 缺点: 学习曲线较陡。
  - **方案 B (Diesel - ORM)**:
    - 优点: 类型安全强。
    - 缺点: 异步支持相对复杂（需要 diesel-async）, 编译耗时极长。
- **决策结论**: **方案 A (SQLx)**。
- **影响**: 确保了 Cowen 核心性能优势，且能原生支持用户要求的多种数据库。

## [ADR-02] 混合模式下的数据一致性策略
- **背景**: 如何确保 Redis 缓存与 DB 持久化之间的数据同步？
- **选型对比**:
  - **方案 A (Write-Through)**: 写 DB 的同时同步写 Redis。
  - **方案 B (Cache Aside)**: 写 DB 后失效 Redis，读时按需加载。
- **决策结论**: **方案 A (Write-Through)**。
- **理由**: 对于 Token 刷新这种低频写、极高频读的场景，Write-Through 能确保更高的实时性，且结构简单。

## [ADR-03] 商店应用长效凭据自愈方案
- **背景**: OAuth2 refresh_token 可能因过期或异常导致失效，若无法自动恢复将导致业务中断。
- **选型对比**:
  - **方案 A (人工干预)**: 报错并要求用户重新扫码授权。
  - **方案 B (永久授权码回溯)**: 在持久化层存储加密的“用户/企业永久授权码”，在 refresh_token 失效时自动换票。
- **决策结论**: **方案 B (永久授权码回溯)**。
- **影响**: 极大提升了商店应用在生产环境下的无人值守能力，符合边车“自愈”的架构愿景。

## [ADR-04] 存储引擎全局绑定 (Global Storage Engine Binding)
- **背景**: 随着 Cowen 从桌面工具向分布式边车演进，原有的按 Profile 粒度指定存储方式（Local vs MySQL）导致了配置管理的复杂性，且在分布式部署时容易产生一致性冲突。
- **决策结论**: **存储方式与应用实例全局绑定**。
- **具体约束**:
  - 存储设置（Storage Type, DB URL 等）从各 Profile 的 `.yaml` 中移出，存入全局配置文件 `app.yaml`。
  - 同一个 Cowen 二进制部署实例下，所有 Profile 必须统一使用相同的物理引擎（要么全部本地磁盘，要么全部远程数据库）。
- **理由**:
  - **降低运维成本**: 生产环境下通常希望所有环境（dev/test/prod）使用统一的物理存储标准，避免管理碎片化。
  - **防止状态孤岛**: 确保全局 Vault 和分布式锁机制能覆盖所有 Profile，真正实现 Sidecar 的水平扩展。
  - **符合直觉**: 存储后端是“基础设施”属性，而非“业务应用”属性。
