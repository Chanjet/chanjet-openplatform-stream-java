# 暂缓与未来演进特性池 (Postponed Features & Future Milestones)

本文档独立于 `v0.1.1` 的核心交付及代码评测范畴。专门用于收口在需求讨论与架构推演中诞生、但评估后认为会导致首发版本“边界失控”、“实施周期冗长”或“复杂度过载”的高阶进阶特性。
这些特性在当下的代码主干分支中无需开发，实体代码将留待后续重大版本迭代时再作考量发力。

---

## 1. 异构平台应用模式兼容扩展 (App Mode Expansion)
> **🏷 暂缓版本：** `v0.1.1`

**提出背景**：除了当前版本重点唯一死保的“企业自建应用”体系外，开放平台宏观生态体系未来还规划接入第三方服务商体系（如 ISV 代开发应用、工具型生态应用等），这类异种构架获取交互票据及证书校验的底层通路截然不同。
**首版妥协点**：全局默认强行绑定唯一下沉模式——`self-built` (自建企业应用)。
**未来实施里程碑期望**：
- 在代码库级提供基础骨架策略抽象 `AppMode (Strategy Pattern)`。
- 支持接收指令级参数 `--app-mode=isv-app` 等模式映射词首，能够在 CLI 运行时通过无缝重写器，分发指向两套甚至多套截然不同源的 `appSecret` 及底座授权机制。

> **🤖 [面向 Agent 研发的实际意义评估]：** 【低 (Low) - 纯运维细节】
> 对于大模型（如 OpenClaw）而言，它作为指令调用方，永远只关心“API 的出入参数据结构长什么样”以及“如何组装语义最快地拿到返回集”。底层究竟是走了自建模式还是第三方的服务商验签网络，对 Agent 的意图推理与任务规划能力毫无增益。唯一有意义的小概率场景是：我们打算塑造一个专门辅助开发者实施“全生态账号体系切换与迁移”的 DevOps 专家大模型。否则，这对普通 Agent 就是一层可有可无的玻璃。

## 2. 微服务分布式泛化存储与并发广播网格 (Distributed Storage SPI)
> **🏷 暂缓版本：** `v0.1.1`

**提出背景**：当企业客户开始利用在类似 Kubernetes (K8s) 等容器化集群编配系统中，以水平无状态的多副本 (ReplicaSets) 形态拉起长链路网关并将其沉降为后台组件时，单机文件系统将面临防腐失效乃至由于宿主 Pod 发生 OOM 驱逐造成的 DLQ 断电死信丢件惨祸。
**首版妥协点**：彻底剔除这套面向分布式的昂贵外设设施桥接代码，纯粹倒退式依赖操作系统的封闭 Keychain 与极轻巧的单机 SQLite。
**未来实施里程碑期望（数据极度冷热分发）：**
- **热电数据分离底盘 SPI 研发**：未来实现针对诸如 `openToken` 的极热瞬态数据强制切入类似 Redis 的中间件进行防并发熔断拦截；并将那些庞大的断点重推账本 (Webhook DLQ) 沉底压入包含重度灾备与检索因子的标量关系库（如 MySQL 或 Elasticsearch）。
- **分布式强状态感知同步网路 (State Sync)**：借由 Redis Pub/Sub 信道实现不同容器间的票据实时共振与死亡内存的热肃清。

> **🤖 [面向 Agent 研发的实际意义评估]：** 【高 (High) - 仅限云原生自治型 Agent】
> - **若 Agent 的定位仅是跑在单机桌面的“本地代码助手 (Copilot)”**：该特性等同废纸，只会带来拖慢开发机性能的噩梦。
> - **若 Agent 是一位部署在服务端、7x24 小时并发工作的“数字员工（自治网络体）”**：那这些特性的意义便是**存亡级的底座**。
>   1. **并发撞库阻击**：服务器端的 Agent 并发思考极快甚至能多副本同时爆发算力。如果没有 Redis 帮其压锁并状态广播共用 Token，上百个 Agent 并行去跟开放平台刷新票据，不仅徒耗算力，更会在第一秒钟内触动开放应用平台的恶暴并发上限阈值（Rate Limit），导致网关被长期封禁断网。
>   2. **深水事故推理能力**：一个高级排障体 Agent 完全可以在夜深人静时，自行通过 API 检索甚至 NL2SQL 连接到位于 MySQL 或 ES 中的全局汇总的死信账簿（DLQ），分析长时序的事件崩溃特征，并向人类运维端提供高度提炼的《网络异常波动自推演报告》。而如果是只存在于单一容器内部的 SQLite 散落文件，一旦 Pod 被回收清洗，这名 Agent 便会由于“没有连贯的数据尸体”，在排障逻辑上彻底沦为瞎子。
